# Spec 042 — LLM provider + per-consumer model config

**Status:** Draft for review (Specify phase)
**Version target:** MINOR (new dashboard-managed provider list + per-consumer model selection;
breaking config-shape change behind a one-time migration; curator/consolidator are opt-in/off by
default, so blast radius is small)
**Depends on:** the existing `llm-connection.ts` secret-key pattern (shipped); the curator/
consolidator config (`curator-config.ts`, shipped)
**Relates to:** `docs/research/self-improving-curator-brainstorm.md` — this spec is feature **2A**,
decision **D3** and §4.4 (all sub-items resolved). 2B (curator unification) and 2C (self-improving
loop) build on it.
**Scope boundary:** **server + dashboard only — no plugin changes.** The five harness plugins never
touch LLM config; this is entirely inside `the-librarian`. (Contrast 1B/spec 041, a six-repo
lockstep.)

---

## Objective

**What.** Replace the single shared `curator.llm.*` connection with **named providers**
(`{ id, name, endpoint, key }`, dashboard-managed CRUD) plus **per-consumer `{ provider, model }`**
selection, so the two LLM-consuming pipelines — the consolidator (intake) and the curator
(grooming) — can run on **different models** (and, if desired, different providers) while reusing
one stored connection. OpenAI-compatible only; a dashboard model-picker auto-populated from
`GET ${endpoint}/models` with a free-text fallback. Embeddings are untouched (bundled-local, not a
provider consumer).

**Why.** Today both pipelines read the same `readCuratorConfig(store).llm`
(`consolidator-tick.ts:37,62`; `curator-tick.ts:43,72`) — one model for both, no way to diverge.
The two jobs have genuinely different cost/latency/quality trade-offs (cheap-fast intake vs.
careful whole-graph grooming), and 2B/2C lean on per-job models. A provider list also lets an admin
register OpenAI, a local Ollama, and an Anthropic-compat endpoint once and point each job wherever
it wants.

**Who.** The self-hosting admin, via the dashboard. No agent-facing surface.

**Success, in one line.** An admin registers two providers in the dashboard, points **intake** at a
cheap model on one and **grooming** at a stronger model on the other, and each pipeline's next run
uses its own provider+model — with API keys stored write-only/encrypted and never round-tripped to
the browser.

## Background — what's there, and the gap (frozen evidence, 2026-06-05)

- **The core LLM client is already OpenAI-compat-only and needs no change.**
  `createCuratorLlmClient` takes `{ endpoint, token, model, timeoutMs }`
  (`curator-llm-client.ts:15-27`), treats `endpoint` as a base URL and appends `/chat/completions`
  (`:102`), uses `Authorization: Bearer` (`:132`), and **always** sets
  `response_format: { type: "json_object" }` (`:116`). There is **no `/models` call anywhere**
  today (the model name is config, not discovered).
- **One shared connection; the two consumers can't diverge.** Both
  `readCuratorConfig` → `config.llm` (`curator-config.ts:55-69`) and the keys are generated by
  `llmConnectionKeys("curator.llm")` → `curator.llm.{provider,endpoint,model,timeout_ms,token}`
  (`llm-connection.ts:57-64`). The consolidator (`consolidator-tick.ts:37,62`) and curator
  (`curator-tick.ts:43,72`) both build their client from that **same** `config.llm`.
- **Keys already use a clean secret pattern — reuse it.** `writeLlmConnection` stores the token via
  `setSetting(keys.token, …, { secret: true })` → AES-256-GCM (`llm-connection.ts:128`);
  `readLlmConnection` exposes **presence only** (`listSettings().some(...)`, `:92`); decryption is
  on-demand via `resolveLlmToken` (`:133-138`). The dashboard token field is write-only/masked,
  never pre-filled (`config-form.tsx:95-102`).
- **Settings store is single-string-per-key.** `SettingsStore` is `{ setSetting, getSetting,
  deleteSetting, listSettings }` with string values and a `{ secret }` flag
  (`settings-types.ts:9-20`); `listSettings` returns metadata, never values. No array/object
  storage today — a provider *list* must be modelled on top of flat keys (see The change §1).
- **Embeddings are independent.** Configured by env only (`LIBRARIAN_EMBEDDER`,
  `LIBRARIAN_MODEL_PATH`), bundled-local, never through `curator.llm.*`
  (`store/index/resolve-embedder.ts:4-43`). Out of scope; the refactor won't touch them.
- **Dead classifier residue.** `classifier-config.ts` does **not** exist; the `classifier`
  references in `llm-connection.ts:2-5,54-55` are stale (classifier removed in spec 032). Clean
  them up as part of this work.
- **Gap.** No provider concept, no per-consumer model, no model discovery. The dashboard's
  `provider`/`endpoint`/`model` are three free-text fields on one connection
  (`config-form.tsx`), and the tRPC surface is `config`/`setConfig`/`runs`/`runNow`
  (`trpc/curator.ts:25-53`).

## The change

### 1. Storage — a provider list on flat secret-aware keys (reuse the proven pattern)

Model a *list* on the existing single-string store rather than redesigning it:

- **Index:** `llm.providers` = JSON array of provider **ids** (non-secret).
- **Per provider `<pid>`:** `llm.provider.<pid>.name`, `llm.provider.<pid>.endpoint` (non-secret),
  `llm.provider.<pid>.token` (**secret**, `{ secret: true }`). Ids are stable + generated; `name`
  is the editable display label (so a rename doesn't break consumer references).
- **Per consumer `<c>`** (`intake`, `grooming` — Decision 1): `curator.<c>.provider` = `<pid>`,
  `curator.<c>.model` = string, `curator.<c>.timeout_ms` = number.
- **Generalise `llm-connection.ts`** so its `read/write/resolve` helpers take an arbitrary
  per-provider prefix (`llm.provider.<pid>`) instead of the hard-wired `curator.llm`. This keeps the
  presence-only-metadata + secret-token-as-separate-key behaviour intact for free.

*Rejected:* one JSON blob in a single `{ secret: true }` setting — it would encrypt the non-secret
names/endpoints too, so the dashboard couldn't list providers without decrypting, and it abandons
the proven presence-only secret pattern.

### 2. Config layer — split the LLM connection out of `CuratorConfig`

- `CuratorConfig.llm` (the single `{ provider, endpoint, model, timeoutMs }`) is **replaced** by
  per-consumer resolution: each consumer resolves `{ provider → endpoint + token, model, timeoutMs
  }` from its `curator.<c>.*` keys + the named provider.
- Leave `promptAddendum`, `enabled`, auto-apply, interval where they are — the addendum's relocation
  to a vault file is **2C's** job (D7), not 2A's. 2A only swaps the connection sub-object.
- Per-consumer operational flags (mirror today's `isLlmComplete`/`isOperational`,
  `curator-config.ts`): a consumer is operational iff its referenced provider exists, has a token,
  and a model is set. A consumer whose provider was deleted reports not-operational (inert, logged),
  never throws.

### 3. Consumers — each builds its client from its own provider + model

- `consolidator-tick.ts` (intake) and `curator-tick.ts` (grooming) each resolve **their** provider
  (endpoint + decrypted token) and **their** model, then build the same `createCuratorLlmClient`
  (unchanged). The shared-config read is gone.

### 4. Dashboard — provider CRUD + per-consumer model selection

- **Provider manager** (new component + tRPC: `listProviders` / `addProvider` / `updateProvider` /
  `deleteProvider` on the admin router): add/edit/delete `{ name, endpoint, key }`; key is
  write-only/masked exactly like today's token (`config-form.tsx:95-102` pattern); deleting a
  provider that a consumer references warns and leaves that consumer not-operational.
- **Per-consumer selector:** for intake and grooming, a provider dropdown + a model field.
- **Model picker:** auto-populate the model dropdown from `GET ${endpoint}/models` (OpenAI-compat
  list endpoint) — **dashboard/tRPC-side call, not the core client**; **free-text fallback** when a
  provider lacks `/models` or errors. This is a UI convenience; nothing at runtime depends on it.

### 5. OpenAI-compat-only — enforce, don't extend

No provider `type`. Every provider speaks OpenAI (`/chat/completions`, Bearer, JSON mode) — the
client already does (§Background). Anthropic is configured via its OpenAI-compat endpoint, not its
native API. **Documented constraint:** the curator relies on `response_format: json_object`; a
compat endpoint that ignores JSON mode will break grooming/intake. Surface this where the admin adds
a provider; the non-blocking "test connection" button (Decision 3) surfaces it without gating.

### 6. Migration — preserve existing installs

On read, if the legacy `curator.llm.*` keys exist and no providers are configured: synthesize one
provider `"default"` from `curator.llm.{endpoint,token}`, point **both** consumers at it with the
legacy `curator.llm.model`, and (one-shot) write the new keys / retire the old. A configured install
keeps working with no admin action; the dashboard then shows one provider + both consumers on it.

### 7. Cleanup

Remove the dead `classifier` references in `llm-connection.ts:2-5,54-55` (spec 032 removed the
classifier; `classifier-config.ts` doesn't exist).

## Commands / Project structure / Testing

- **Touches (server):** `packages/core/src/llm-connection.ts` (generalise prefix; drop classifier
  residue), `curator-config.ts` (split out the connection; per-consumer resolution + operational
  flags), `consolidator-tick.ts` + `curator-tick.ts` (per-consumer build), a new provider-store
  module (`llm-providers.ts` or similar), `packages/mcp-server/src/trpc/curator.ts` (provider CRUD +
  a `listModels(endpoint)` query for the picker).
- **Touches (dashboard):** `apps/dashboard/components/curator/config-form.tsx` + a new provider-
  manager component.
- **Untouched:** `curator-llm-client.ts` (core client), embeddings (`resolve-embedder.ts`), the five
  plugins, the prompt addendum's storage (2C).
- **Testing:** unit — provider CRUD round-trips (token write-only, presence-only metadata, encrypted
  at rest); per-consumer resolution picks the right provider+model; a deleted provider →
  consumer not-operational (no throw); the legacy-key **migration** synthesises `"default"` and both
  consumers resolve to it; JSON-mode request unchanged. tRPC — `listModels` parses an OpenAI-compat
  `/models` response and falls back cleanly on error/timeout. Dashboard — provider add/edit/delete;
  model dropdown populates and falls back to free-text. No secret literals in fixtures (assemble at
  runtime; see AGENTS.md GitGuardian note).
- **CHANGELOG:** one `## [Unreleased]` entry (single repo).

## Boundaries

- **Always:** OpenAI-compat-only (no provider `type`); keys write-only/masked, encrypted at rest,
  never round-tripped or logged; per-consumer resolution fail-soft (missing/deleted provider →
  inert, never throws); reuse the existing secret-key + presence-only pattern; branch + PR;
  CHANGELOG.
- **Out of scope:** the curator **unification** / job renaming narrative (2B); the addendum moving to
  a vault file + the self-improvement loop (2C, D7/D8); any change to the core LLM client, the
  embeddings path, or native (non-compat) provider APIs.
- **Never:** make embeddings a provider consumer (D3 §4.4b — stays bundled-local, zero-config);
  store a provider key anywhere non-secret; expose any of this to the agent-facing MCP surface.

## Success criteria

- [ ] An admin can add/edit/delete named providers `{ name, endpoint, key }` in the dashboard; keys
  are write-only/masked and encrypted at rest (presence-only in metadata).
- [ ] Intake and grooming each select their own `{ provider, model }`; a run uses the selected one;
  the two can differ (verified by a test that points them at different providers/models).
- [ ] The model dropdown auto-populates from `GET ${endpoint}/models` and falls back to free-text on
  error/timeout; the **core LLM client makes no `/models` call**.
- [ ] Deleting a provider a consumer references leaves that consumer not-operational (inert, logged),
  never throws.
- [ ] A pre-existing `curator.llm.*` install migrates to one `"default"` provider with both consumers
  pointed at it, with **no admin action** and no behaviour change.
- [ ] Embeddings are unchanged; the core LLM client is unchanged; no plugin is touched.
- [ ] The dead `classifier` references in `llm-connection.ts` are gone.
- [ ] No secret ever appears in logs, error messages, tRPC responses, or the browser.

## Decisions (settled — build-ready)

All four are settled; **nothing here blocks an autonomous build.**

1. **Consumer names = `intake` / `grooming`** (owner, 2026-06-05). The two per-consumer config
   groups use the D2 target names now, so 2B's unification doesn't re-key the persisted settings.
   Setting keys: **`curator.intake.*`** (the consolidator pipeline) and **`curator.grooming.*`**
   (the curator pipeline) — consistent with D2's "the curator" entity with two jobs.
2. **Provider key = a generated stable `id`, with an editable display `name`.** Consumer references
   (`curator.<job>.provider`) point at the `id`, so renaming a provider never breaks a reference.
3. **Provider validation = a non-blocking "test connection" button, not a hard gate.** It runs a
   cheap check (a `/models` hit, or a tiny JSON-mode completion) and surfaces the result; **adding a
   provider never blocks on it.** This catches endpoints that ignore `response_format: json_object`
   (which would silently break a job) but leaves the admin to heed the warning.
4. **Per-consumer `timeout_ms`.** Each consumer carries its own timeout (grooming may want longer
   than intake); no shared value.
